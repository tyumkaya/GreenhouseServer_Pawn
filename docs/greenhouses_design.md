# Greenhouses Design

Дата: 2026-04-16

Документ описывает проектирование системы теплиц для текущего open.mp / Pawn проекта без реализации кода на этом шаге.

## 1. Цели и ограничения

- До 5 теплиц на одного игрока.
- Рост идет только пока владелец онлайн.
- Полное созревание растения: 10 минут базово.
- Улучшение теплицы может ускорять рост в 2 раза.
- Взаимодействие только рядом с объектом/точкой теплицы.
- Визуализация идет стадиями, без постоянного пересоздания объектов.
- Сохранение должно избегать лишних `UPDATE`.
- Архитектура должна держать около 1000 онлайна.

## 2. Модульная структура

Рекомендуемый новый модуль:

- `server/greenhouses/variables.inc`
- `server/greenhouses/public.inc`
- `server/greenhouses/mysql.inc`
- `server/greenhouses/stock.inc`
- `server/greenhouses/commands.inc`
- `server/greenhouses/dialog.inc`
- `server/greenhouses/dialog_response.inc`

Назначение:

- `variables.inc`: все enum/массивы/runtime state.
- `public.inc`: публичные callbacks и lifecycle hooks.
- `mysql.inc`: callbacks после загрузки/сохранения.
- `stock.inc`: внутренняя логика, проверки, расчеты роста, dirty-marking.
- `commands.inc`: временные админ-команды и игровые команды.
- `dialog*.inc`: UI при необходимости.

## 3. Структура данных в памяти

### 3.1. Глобальная структура теплицы

Нужен единый пул теплиц сервера, а не массив "по игроку", чтобы удобно грузить из БД, искать по id и обновлять централизованно.

Рекомендуемая структура:

- `ghSQLID`
- `ghOwnerID`
- `ghSlot`
- `ghType`
- `ghUpgradeLevel`
- `ghGrowthMultiplier`
- `ghState`
- `ghPlantType`
- `ghStage`
- `ghGrowthSeconds`
- `ghMatureAtGrowthSeconds`
- `ghLastPersistGrowthSeconds`
- `ghLastPersistStage`
- `ghDirtyFlags`
- `ghLoaded`
- `ghOwnerOnline`
- `ghPosX`
- `ghPosY`
- `ghPosZ`
- `ghAngle`
- `ghWorld`
- `ghInterior`
- `ghObjectSeedling`
- `ghObjectGrowing`
- `ghObjectMature`
- `ghLabelID`
- `ghAreaID`
- `ghLastInteractionTick`

Смысл полей:

- `ghSQLID`: первичный ключ строки в БД.
- `ghOwnerID`: ссылка на `users.id`.
- `ghSlot`: индекс теплицы игрока от 0 до 4.
- `ghType`: тип теплицы, если появятся разные классы.
- `ghUpgradeLevel`: уровень улучшения.
- `ghGrowthMultiplier`: множитель роста, например `1` или `2`.
- `ghState`: пустая, посажена, растет, созрела, заблокирована.
- `ghPlantType`: тип растения.
- `ghStage`: текущая визуальная стадия.
- `ghGrowthSeconds`: накопленное время роста только в онлайне владельца.
- `ghMatureAtGrowthSeconds`: цель для созревания, базово `600`.
- `ghLastPersistGrowthSeconds`: что уже сохранено в БД.
- `ghLastPersistStage`: последняя сохраненная стадия.
- `ghDirtyFlags`: что именно изменилось.
- `ghLoaded`: теплица загружена в память.
- `ghOwnerOnline`: владелец сейчас онлайн.
- координаты/world/interior: для streamer.
- object/label/area ids: runtime ids streamer.
- `ghLastInteractionTick`: антиспам на взаимодействие.

### 3.2. Player runtime структура

Нужен минимальный runtime state на игрока:

- `pUserSQLID`
- `pAuthorized`
- `pGreenhouseLoaded`
- `pGreenhouseCount`
- `pGreenhouseIDs[5]`
- `pNearbyGreenhouseID`

Назначение:

- `pUserSQLID`: реальный `users.id`, его нужно начать загружать при логине.
- `pAuthorized`: лучше отдельный флаг post-auth, а не только текущий `pLogin`.
- `pGreenhouseLoaded`: теплицы игрока загружены в runtime.
- `pGreenhouseCount`: сколько теплиц привязано.
- `pGreenhouseIDs[5]`: ссылки на глобальные индексы теплиц.
- `pNearbyGreenhouseID`: ближайшая доступная теплица для взаимодействия.

### 3.3. Dirty flags

Чтобы не делать лишние `UPDATE`, у каждой теплицы лучше хранить битовую маску:

- `GH_DIRTY_NONE`
- `GH_DIRTY_STATE`
- `GH_DIRTY_GROWTH`
- `GH_DIRTY_STAGE`
- `GH_DIRTY_UPGRADE`
- `GH_DIRTY_POSITION`
- `GH_DIRTY_OWNER`

Тогда сохранение делает только нужный `UPDATE` и только если реально что-то менялось.

## 4. SQL-схема

### 4.1. Основная таблица

```sql
CREATE TABLE `greenhouses` (
    `id` INT UNSIGNED NOT NULL AUTO_INCREMENT,
    `owner_id` INT UNSIGNED NOT NULL,
    `slot` TINYINT UNSIGNED NOT NULL,
    `type` TINYINT UNSIGNED NOT NULL DEFAULT 1,
    `upgrade_level` TINYINT UNSIGNED NOT NULL DEFAULT 0,
    `growth_multiplier` TINYINT UNSIGNED NOT NULL DEFAULT 1,
    `state` TINYINT UNSIGNED NOT NULL DEFAULT 0,
    `plant_type` TINYINT UNSIGNED NOT NULL DEFAULT 0,
    `stage` TINYINT UNSIGNED NOT NULL DEFAULT 0,
    `growth_seconds` SMALLINT UNSIGNED NOT NULL DEFAULT 0,
    `mature_at_growth_seconds` SMALLINT UNSIGNED NOT NULL DEFAULT 600,
    `pos_x` FLOAT NOT NULL,
    `pos_y` FLOAT NOT NULL,
    `pos_z` FLOAT NOT NULL,
    `angle` FLOAT NOT NULL DEFAULT 0,
    `world` INT NOT NULL DEFAULT 0,
    `interior` INT NOT NULL DEFAULT 0,
    `created_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    UNIQUE KEY `uniq_owner_slot` (`owner_id`, `slot`),
    KEY `idx_owner_id` (`owner_id`)
);
```

### 4.2. Почему одной таблицы достаточно на старте

На текущем этапе достаточно одной таблицы `greenhouses`, потому что:

- ограничение маленькое: максимум 5 на игрока;
- по описанию одна теплица = один растущий объект состояния;
- нет требования к нескольким грядкам внутри одной теплицы;
- меньше SQL, меньше join-ов, проще lifecycle.

Если позже внутри одной теплицы появится несколько посадочных мест, тогда схема расширяется на:

- `greenhouse_units`
- `greenhouse_access`
- `greenhouse_logs`

Но сейчас это преждевременно.

### 4.3. Связь с users

`owner_id` должен ссылаться на `users.id`.

Критически важно:

- при логине надо начать реально загружать `users.id` в память игрока;
- без этого подсистема теплиц не должна строиться.

## 5. Состояния и стадии

### 5.1. Состояния теплицы

Предлагаемые состояния:

- `GH_STATE_EMPTY`
- `GH_STATE_PLANTED`
- `GH_STATE_GROWING`
- `GH_STATE_MATURE`

### 5.2. Стадии визуализации

Для визуала лучше 4 стадии:

- `GH_STAGE_EMPTY`
- `GH_STAGE_1`
- `GH_STAGE_2`
- `GH_STAGE_3_MATURE`

Логика переходов:

- `0%` -> пусто
- `1% - 33%` -> стадия 1
- `34% - 66%` -> стадия 2
- `67% - 99%` -> стадия 3 почти зрелая
- `100%` -> зрелая стадия

На практике можно оставить и 3 стадии роста плюс отдельную зрелую, если найдутся удобные GTA-объекты.

## 6. Визуализация

### 6.1. Через streamer

Использовать уже установленный `streamer`:

- `CreateDynamicObject`
- `CreateDynamic3DTextLabel`
- `CreateDynamicArea`

### 6.2. Как визуализировать без лишней нагрузки

Не надо постоянно удалять и создавать объекты при каждом тике роста.

Правильная стратегия:

- при загрузке теплицы создать фиксированный runtime-набор streamer entities;
- менять визуал только при переходе между стадиями;
- если stage не изменился, ничего с объектами не делать;
- для каждой теплицы держать только:
  - 1 основной dynamic object
  - 1 dynamic area
  - 1 optional 3D text label

Еще лучше:

- один dynamic object, у которого меняется модель/позиция только в момент смены стадии;
- label обновлять только при заметных изменениях:
  - посадка
  - смена стадии
  - созревание
  - сбор урожая

То есть не обновлять текст каждую секунду.

## 7. Взаимодействие только рядом

### 7.1. Проверка доступа

Игрок может взаимодействовать с теплицей только если:

- авторизован;
- владелец этой теплицы;
- находится в правильном `world` и `interior`;
- находится в radius около точки входа/объекта;
- не упирается в cooldown антиспама.

### 7.2. Практический способ

Использовать `streamer area` для грубой подписки и дополнительную проверку расстояния при действии.

Причина:

- area дешево говорит, что игрок рядом;
- финальная `GetPlayerPos + distance` перед действием защищает от граничных случаев.

## 8. Рост только когда владелец онлайн

### 8.1. Основной принцип

Рост не должен зависеть от real-time оффлайн.

Правильная модель:

- у теплицы хранится `growth_seconds`;
- пока владелец онлайн, это значение растет;
- как только владелец вышел, рост останавливается;
- при следующем логине продолжается с сохраненного значения.

### 8.2. Формула роста

Базовое созревание:

- `mature_at_growth_seconds = 600`

Базовый апдейт:

- без улучшения: `+1 growth second` за 1 серверную секунду
- с x2 улучшением: `+2 growth seconds` за 1 серверную секунду

То есть:

- без улучшения полное созревание за 10 минут онлайна владельца;
- с улучшением x2 полное созревание за 5 минут онлайна владельца.

### 8.3. Как обновлять

Не делать таймер на каждую теплицу.

Нужен один глобальный heartbeat, например раз в 1 секунду:

- пройти по списку активных теплиц;
- если владелец онлайн и теплица в состоянии роста, увеличить `growth_seconds`;
- пересчитать `stage`;
- если стадия изменилась, обновить визуал и пометить dirty;
- если рост дошел до `600`, перевести в `MATURE`.

Это соответствует требованию оптимизации под высокий онлайн.

## 9. Жизненный цикл

### 9.1. Создание теплицы

Сценарий:

1. Проверить, что игрок авторизован.
2. Проверить, что у игрока меньше 5 теплиц.
3. Найти свободный `slot` от 0 до 4.
4. Зафиксировать координаты/мир/интерьер.
5. Вставить строку в `greenhouses`.
6. Создать runtime-сущность в памяти.
7. Поднять streamer object/area/label.
8. Добавить ссылку в `pGreenhouseIDs`.

### 9.2. Загрузка при логине

Сценарий:

1. Игрок проходит auth.
2. Должен быть известен его `users.id`.
3. Выполняется один `SELECT * FROM greenhouses WHERE owner_id = ? ORDER BY slot ASC`.
4. Все найденные теплицы загружаются в глобальный пул.
5. Строятся streamer entities.
6. В `pGreenhouseIDs[5]` записываются runtime ids.
7. `pGreenhouseLoaded = true`.

Важно:

- эту загрузку лучше делать не в `OnPlayerConnect`, а только после полной авторизации;
- иначе теплицы можно случайно открыть до логина.

### 9.3. Работа в онлайне

Пока игрок онлайн:

- глобальный heartbeat обновляет только его активные теплицы;
- dirty changes копятся в памяти;
- сохранение идет отложенно.

### 9.4. Выгрузка при дисконнекте

Сценарий:

1. Остановить рост, пометив владельца как offline.
2. Принудительно выполнить flush dirty-данных игрока по его теплицам.
3. Удалить streamer entities этих теплиц.
4. Освободить runtime-слоты из памяти.
5. Очистить `pGreenhouseIDs`, `pNearbyGreenhouseID`, `pGreenhouseLoaded`.

Почему так:

- рост идет только онлайн;
- держать оффлайн-теплицы в памяти нет смысла;
- это экономит память и streamer entities при 1000 онлайна.

## 10. Стратегия сохранения без лишних UPDATE

### 10.1. Чего нельзя делать

Нельзя:

- писать `UPDATE` каждую секунду;
- писать `UPDATE` при каждом тике;
- писать `UPDATE` только ради небольшого изменения текста/визуала.

### 10.2. Правильная стратегия

Сохранять в БД только по событиям:

- создание теплицы
- посадка
- смена стадии
- улучшение
- сбор урожая
- дисконнект владельца
- периодический flush раз в 60 секунд для dirty-теплиц

### 10.3. Что хранить только в памяти

В памяти можно держать:

- текущие streamer ids
- промежуточный секундый прогресс между flush
- nearby state игрока

### 10.4. Что писать в БД

В БД писать:

- `state`
- `plant_type`
- `stage`
- `growth_seconds`
- `upgrade_level`
- `growth_multiplier`
- при необходимости координаты

### 10.5. Flush policy

Рекомендуемая политика:

- mark dirty при значимом изменении;
- global save queue;
- batch flush раз в 60 секунд;
- force flush при disconnect;
- force flush при stage transition;
- force flush при mature state.

Это сильно режет SQL-нагрузку и при этом не теряет много прогресса.

## 11. Оптимизация под 1000 онлайна

### 11.1. Главные принципы

- один глобальный update timer на систему, не таймер на теплицу;
- обновлять только активные теплицы онлайн-владельцев;
- не держать оффлайн-теплицы в runtime;
- не делать постоянный rebuild streamer entities;
- не делать постоянные SQL `UPDATE`;
- хранить быстрые ссылки `player -> greenhouse ids`.

### 11.2. Оценка нагрузки

Худший разумный сценарий:

- 1000 игроков онлайн;
- по 5 теплиц;
- максимум 5000 активных теплиц.

Это много, но все еще управляемо, если:

- heartbeat 1 раз в секунду;
- логика на теплицу очень дешевая;
- streamer объекты меняются только на stage change;
- SQL идет пакетно и редко.

### 11.3. Как сократить CPU

- не считать расстояние до всех теплиц для всех игроков каждый тик;
- использовать dynamic area callbacks, а не полный перебор;
- не пересчитывать stage, если теплица уже `MATURE`;
- не обновлять label каждую секунду;
- по возможности хранить активный список "растущих теплиц", а не обходить весь пул.

### 11.4. Активный список

Лучший вариант:

- отдельный массив/список активных теплиц;
- в нем только те, что:
  - загружены
  - владелец онлайн
  - состояние `PLANTED/GROWING`

Тогда heartbeat работает только по активному списку.

### 11.5. SQL-оптимизация

- один `SELECT` теплиц на логин владельца;
- один `INSERT` при создании;
- отложенные `UPDATE` при dirty flush;
- без join-ов и без постоянных выборок.

## 12. Рекомендуемые игровые правила на уровне системы

- Игрок может иметь не более 5 теплиц.
- Каждая теплица занимает фиксированный слот.
- Взаимодействие только рядом с area/объектом.
- Улучшение x2 меняет только скорость роста, не меняя SQL-логику.
- Созревшее растение не продолжает обновляться.
- После сбора урожая теплица переходит в `EMPTY`.

## 13. Что надо изменить в текущем проекте перед реализацией

### Обязательно

- Начать реально загружать `users.id` в player runtime.
- Ввести единый post-auth flow:
  - общий хук после успешного логина
  - и такой же хук после регистрации
- Добавить новый модуль `server/greenhouses/*`.

### Желательно

- Явно развести:
  - account auth state
  - gameplay-ready state
- Это упростит безопасную загрузку теплиц после логина.

## 14. Точный список ручных действий для вас

1. Подтвердить, что одна теплица на старте содержит один растущий объект, а не несколько грядок внутри.
2. Подтвердить список культур или разрешить старт с одним типом растения.
3. Подтвердить, что улучшение пока только одно: `x2 growth speed`.
4. Подтвердить, что ограничение ровно `5` теплиц на игрока и оно не зависит от VIP/админ-статуса.
5. Подтвердить, что теплицы доступны только владельцу, без shared access на первом этапе.
6. Подтвердить, что рост должен идти только при онлайне владельца, без оффлайн-прогресса.
7. Подтвердить, что можно использовать `streamer` как основную систему объектов, зон и 3D-текста.
8. Применить будущую SQL-миграцию в вашей MySQL БД после того, как я подготовлю ее на следующем шаге.
9. Проверить, что в вашей БД существует и используется таблица `users` с корректным `id`, на который можно ссылаться из `greenhouses.owner_id`.
10. Если у вас есть нестандартный compile command вне Qawno, прислать его перед этапом реализации.
